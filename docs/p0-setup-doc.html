SENTRY — P0-17 — 3 BUGS SUCCESSIFS RÉSOLUS
BUG 1 — APP_FILTER string au lieu du token NestJS
provide: "APP_FILTER" (string) → NestJS enregistre un provider arbitraire, jamais appliqué comme filtre global.
Fix : provide: APP_FILTER (token importé de @nestjs/core).
BUG 2 — TIMING DOTENV DANS instrument.ts
instrument.ts s'exécute avant @nestjs/config → process.env["SENTRY_DSN"] est undefined → enabled: false → Sentry désactivé silencieusement, aucun event envoyé.
Fix : ajouter dotenv.config() en premier dans instrument.ts, avant Sentry.init(). dotenv est déjà dans les devDependencies.
BUG 3 — SentryGlobalFilter INCOMPATIBLE AVEC PINO MULTI-TRANSPORT EN PRODUCTION
buildPinoConfig en production utilise targets (multi-transport) → worker threads pour pino/file + @axiomhq/pino. Ces worker threads interfèrent avec AsyncLocalStorage d'OpenTelemetry (utilisé par Sentry) → contexte async perdu → events créés mais droppés silencieusement.
SentryGlobalFilter fonctionne en dev (pino-pretty = thread principal), pas en production.
Sentry.captureException() manuel fonctionne partout car pas de dépendance au contexte async.
Fix : SentryExceptionFilter custom (src/filters/sentry-exception.filter.ts) étendant BaseExceptionFilter, appelle Sentry.captureException() directement. Ne capture pas les HttpException (4xx attendus) — seulement les erreurs non-HTTP (vrais 500).
ESLINT — eslint-plugin-unicorn NON INSTALLÉ
eslint-plugin-unicorn n'est pas installé dans ce projet. Référencer // eslint-disable-next-line unicorn/prefer-top-level-await = erreur ESLint au commit (pre-commit hook husky).
Erreur symptomatique : "Definition for rule 'unicorn/prefer-top-level-await' was not found."
Fix : supprimer le commentaire eslint-disable. La solution finale pour main.ts est bootstrap().catch((_: unknown) => {'{ process.exit(1); }'}) sans aucun commentaire de disable.
